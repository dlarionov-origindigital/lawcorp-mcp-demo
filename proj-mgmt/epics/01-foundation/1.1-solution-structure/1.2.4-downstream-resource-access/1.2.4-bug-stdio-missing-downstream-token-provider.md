# BUG: IDownstreamTokenProvider not registered on stdio transport when UseAuth=true

**Status:** DONE
**Type:** Bug
**Severity:** Blocker (prevents server startup)
**Story:** [1.2.4: Downstream resource access via identity passthrough](./1.2.4-downstream-resource-access.md)
**Discovered:** 2026-03-10
**Trigger:** .NET 10 upgrade (ValidateOnBuild now enabled by default)

---

## Symptom

When starting the MCP server after upgrading to .NET 10, the app crashes immediately at `builder.Build()` with:

```
System.AggregateException: Some services are not able to be constructed
  (Error while validating the service descriptor
   'ServiceType: MediatR.IRequestHandler`2[...GetDocumentByIdQuery,...GetDocumentResult]
    Lifetime: Transient
    ImplementationType: ...GetDocumentByIdHandler':
    Unable to resolve service for type 'LawCorp.Mcp.Core.Auth.IDownstreamTokenProvider'
    while attempting to activate '...GetDocumentByIdHandler'.)
```

All three document handlers fail: `GetDocumentByIdHandler`, `ListDocumentsByCaseHandler`, `SearchDocumentsHandler`.

---

## Root Cause

`RunStdioAsync` in `ServerBootstrap.cs` does not register `IDownstreamTokenProvider` when `UseAuth=true`.

**The registration gap:**

| Transport | UseAuth | Who registers `IDownstreamTokenProvider`? | Result |
|---|---|---|---|
| HTTP | `false` | `RegisterSharedServices` → `NoOpTokenProvider` (singleton) | ✅ |
| HTTP | `true` | `ConfigureAuth` → `AddEntraIdAuthentication` → `OboDownstreamTokenProvider` (scoped) | ✅ |
| Stdio | `false` | `RegisterSharedServices` → `NoOpTokenProvider` (singleton) | ✅ |
| **Stdio** | **`true`** | **Nobody** | ❌ |

The stdio path (`RunStdioAsync`) deliberately falls back to `AnonymousUserContext` when `UseAuth=true` (with a warning), but forgets to also fall back to `NoOpTokenProvider`. Since `RegisterSharedServices` guards the `NoOpTokenProvider` registration behind `if (!UseAuth)`, and the stdio path never calls `ConfigureAuth` (which would call `AddEntraIdAuthentication`), there is no code path that registers `IDownstreamTokenProvider`.

### Why this surfaced now

In .NET 9, `Host.CreateApplicationBuilder()` did **not** enable `ValidateOnBuild` by default. The missing registration was latent — it would only surface at runtime when a document handler was actually invoked via MCP `tools/call`. In .NET 10, `Host.CreateApplicationBuilder()` enables `ValidateOnBuild = true` by default, catching all unresolvable service chains eagerly at startup.

If the user was previously running HTTP transport (which is the `appsettings.json` default), the bug in the stdio path was never exercised. The .NET 10 upgrade + switching to stdio (or running the DI validation in any context) exposes it.

### Additional risk: HTTP transport

If `AddEntraIdAuthentication` throws during the Microsoft.Identity.Web registration chain before reaching the `AddScoped<IDownstreamTokenProvider>` line (line 58 of `AuthServiceCollectionExtensions.cs`), the same symptom would appear on the HTTP path too. This could happen if `Microsoft.Identity.Web 4.3.0` has compatibility issues with .NET 10. A defensive fallback would protect against this.

---

## Code References

- `ServerBootstrap.RunStdioAsync` — registers `AnonymousUserContext` but not `NoOpTokenProvider`
- `ServerBootstrap.RegisterSharedServices` — guards `NoOpTokenProvider` behind `if (!UseAuth)`
- `ServerBootstrap.ConfigureAuth` → `AddEntraIdAuthentication` — only called from HTTP path
- `AuthServiceCollectionExtensions.AddEntraIdAuthentication` line 58 — `AddScoped<IDownstreamTokenProvider, OboDownstreamTokenProvider>`

---

## Recommended Fix

Register `NoOpTokenProvider` in `RunStdioAsync` explicitly, consistent with the existing `AnonymousUserContext` fallback:

```csharp
// In RunStdioAsync, after the UseAuth warning block:
builder.Services.AddSingleton<IUserContext, AnonymousUserContext>();
builder.Services.AddSingleton<IDownstreamTokenProvider, NoOpTokenProvider>();  // ADD THIS
RegisterSharedServices(builder.Services, builder.Configuration);
```

This ensures stdio always has a no-op implementation, matching the existing pattern for `IUserContext`. The subsequent `if (!UseAuth)` guard in `RegisterSharedServices` becomes a harmless duplicate registration (last-wins).

### Alternative (more defensive)

Move the `NoOpTokenProvider` registration out of the `UseAuth` guard in `RegisterSharedServices` and use `TryAddSingleton` so it only takes effect when no other implementation is registered:

```csharp
// In RegisterSharedServices, replace the conditional block:
services.TryAddSingleton<IDownstreamTokenProvider, NoOpTokenProvider>();
```

This ensures every transport/auth combination has a fallback, while `AddEntraIdAuthentication`'s scoped `OboDownstreamTokenProvider` takes precedence when auth is active (scoped overrides singleton in resolution scope).
